# dotclaude

Portable Claude Code configuration — global `CLAUDE.md`, skills, and agents — versioned in one repo and symlinked into `~/.claude/` on each machine.

Clone once, run `install.sh`, and your full Claude setup is live. Pull to sync updates across machines.

---

## What's included

| Path | Description |
|---|---|
| `CLAUDE.md` | Global standing instructions for every Claude Code session |
| `settings.json` | Preferences: model, theme, effort level, status line |
| `statusline-command.sh` | Custom terminal status line script |
| `skills/` | 8 global skills (see below) |
| `agents/` | Custom agents |

### Skills

| Skill | Trigger | Purpose |
|---|---|---|
| `adr` | `/adr` | Write an Architecture Decision Record |
| `diagnosing-bugs` | `/diagnosing-bugs` | 6-phase debugging protocol — feedback loop first, hypothesis second |
| `domain-modeling` | `/domain-modeling` | Build and maintain a project glossary (`CONTEXT.md`) and ADRs |
| `graphify` | `/graphify` | Turn a codebase into a queryable knowledge graph |
| `grill-with-docs` | `/grill-with-docs` | Relentless design interview — surfaces assumptions, writes ADRs as decisions crystallize |
| `prd` | `/prd` | Write or update a Product Requirements Document |
| `tdd` | `/tdd` | Test-driven development via vertical slices (one test → one impl → repeat) |
| `tech-spec` | `/tech-spec` | Write a technical design doc / RFC before building |

### Agents

| Agent | Purpose |
|---|---|
| `scale-reviewer` | Read-only scalability reviewer — catches N+1s, O(history) hot paths, unbounded queries at ~100× current data |

## Setup — new machine

### macOS / Linux

```bash
git clone git@github.com:Dreynaldis/dotclaude.git ~/dotclaude
bash ~/dotclaude/install.sh
source ~/.zshrc   # or ~/.bashrc depending on your shell
```

No prerequisites. `ln -s` works for files and directories without special permissions.

### Windows (Git Bash)

1. Enable **Developer Mode**: Settings → System → Developer Mode → On
   _(required for file symlinks — directory junctions work without it)_
2. Clone and install:

```bash
git clone git@github.com:Dreynaldis/dotclaude.git D:/project/dotclaude
bash D:/project/dotclaude/install.sh
source ~/.bashrc
```

`install.sh` handles both cases — if Developer Mode is off, files are copied with a warning and you can re-run after enabling it.

### What install.sh does

- Backs up any existing `~/.claude/` files it's about to replace (`.bak` suffix)
- Creates symlinks/junctions from `~/.claude/` → this repo
- Injects a `dotclaude` alias into your shell rc file for quick access

## Daily workflow

### Updating config (edit a skill, tweak CLAUDE.md, etc.)

Edits via `~/.claude/` and `<repo>/` are the same files — pick either path.

```bash
# Then commit and push from the repo:
dotclaude
git add .
git commit -m "..."
git push
```

### Pulling updates on another machine

```bash
dotclaude
git pull
# Done — symlinks mean changes are live immediately, no reinstall needed
```

### Adding a new skill

```bash
mkdir ~/.claude/skills/my-skill   # junction writes directly into repo
# create ~/.claude/skills/my-skill/SKILL.md

dotclaude
git add .
git commit -m "add: my-skill"
git push
```

## Symlink map

After install, `~/.claude/` contains:

```
~/.claude/
  CLAUDE.md             → <repo>/CLAUDE.md              (SymbolicLink)
  settings.json         → <repo>/settings.json          (SymbolicLink)
  statusline-command.sh → <repo>/statusline-command.sh  (SymbolicLink)
  skills/               → <repo>/skills/                (Junction on Windows, symlink on Unix)
  agents/               → <repo>/agents/                (Junction on Windows, symlink on Unix)
  ... (session data, cache, etc. — not tracked)
```

Claude Code reads `~/.claude/` as normal. It has no knowledge of the repo.
